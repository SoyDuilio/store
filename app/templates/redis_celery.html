Redis y Celery en el Sistema ERP

🔴 REDIS - Base de Datos en Memoria

¿Qué es Redis?

Redis (Remote Dictionary Server) es una base de datos que almacena información en la memoria RAM en lugar del disco duro. Esto la hace extremadamente rápida: consultas que en PostgreSQL tardan 500ms, en Redis tardan 2ms.

Piensa en Redis como un "diccionario Python gigante compartido en la nube" al que todas las instancias de tu aplicación pueden acceder simultáneamente.

1. Sesión Única de Usuario (Requerimiento Crítico)

❌ Sin Redis

Problema: Cada instancia guarda sesiones en su propia memoria.

1
Usuario inicia sesión en servidor A → token guardado en memoria local de A
2
Usuario abre otra pestaña → conecta a servidor B
3
Servidor B no sabe que ya hay sesión activa

⚠️ Resultado: Usuario puede tener múltiples sesiones simultáneas

✅ Con Redis

Solución: Redis es compartido entre todas las instancias.

async def login(user_id: int, token: str): # Verificar si ya hay sesión activa active_token = await redis.get(f"session:{user_id}") if active_token and active_token != token: raise HTTPException(401, "Ya hay sesión activa en otro dispositivo") # Guardar nueva sesión (expira en 1 hora) await redis.setex(f"session:{user_id}", 3600, token)

✓ Garantiza: Solo una sesión activa por usuario

Escenario: El catálogo de productos se consulta en cada venta del POS.

❌ Sin Cache (cada consulta a PostgreSQL)

def get_productos(): # Cada llamada consulta la base de datos return db.query(Producto).all() # ⏱️ 500ms cada vez

3. Rate Limiting (Prevenir Abuso)

Protección contra ataques o uso excesivo

# Limitar 100 requests por minuto por usuario async def check_rate_limit(user_id: int): key = f"ratelimit:{user_id}:{datetime.now().minute}" count = await redis.incr(key) await redis.expire(key, 60) if count > 100: raise HTTPException(429, "Demasiadas peticiones")

Redis incrementa un contador por usuario por minuto. Si supera el límite, rechaza las peticiones automáticamente.

🌾 CELERY - Sistema de Tareas Asíncronas

¿Qué es Celery?

Celery es un sistema de colas de tareas que permite ejecutar trabajos pesados en segundo plano sin bloquear la respuesta al usuario.

Imagina una cajera en un banco: cuando depositas dinero, no esperas a que el cajero cuente cada billete, lo guarde en la bóveda, actualice el sistema central, etc. El cajero te da un comprobante inmediatamente y hace todo lo demás después.

Celery hace exactamente eso con tu sistema.

1. Facturación Electrónica (Caso Crítico para tu Sistema)

🚨 El Problema Sin Celery

La cajera hace clic en "Facturar" y debe esperar a que TODO el proceso termine antes de poder atender al siguiente cliente.

❌ Sin Celery (Bloqueante)

@router.post("/facturar") async def crear_factura(data: FacturaCreate): factura = crear_factura_db(data) # 100ms xml = generar_xml(factura) # 200ms firmar_xml(xml) # 300ms response = enviar_sunat(xml) # 2000ms ⚠️ guardar_cdr(response) # 100ms return factura # ⏱️ TIEMPO TOTAL: 2,700ms = 2.7 segundos

Peor escenario: Si SUNAT está lento o falla:

  • Timeout después de 30 segundos

✅ Con Celery (Asíncrono)

@router.post("/facturar") async def crear_factura(data: FacturaCreate): factura = crear_factura_db(data) # 100ms # Enviar tarea a segundo plano enviar_factura_sunat.delay(factura.id) return factura # ⚡ Responde INMEDIATAMENTE # ⏱️ TIEMPO DE RESPUESTA: 100ms
# Tarea Celery (corre en segundo plano) @celery_app.task(bind=True, max_retries=3) def enviar_factura_sunat(self, factura_id: int): try: factura = db.get(factura_id) xml = generar_xml(factura) firmar_xml(xml) response = enviar_sunat(xml) guardar_cdr(response) # Notificar via WebSocket que terminó notify_user(factura.user_id, "✅ Factura enviada a SUNAT") except Exception as e: # Reintentar después de 1 minuto raise self.retry(exc=e, countdown=60)

2. Reintentos Automáticos (Red Inestable / SUNAT Caído)

Escenario Real en Perú

SUNAT tiene caídas frecuentes, especialmente fin de mes. Sin Celery, pierdes ventas o tienes que rehacer facturas manualmente.

@celery_app.task( bind=True, max_retries=10, # Reintentar hasta 10 veces default_retry_delay=300 # Cada 5 minutos ) def enviar_factura_sunat(self, factura_id): try: response = enviar_a_sunat(factura_id) return response except SUNATException as e: # Si SUNAT responde con error técnico, reintentar if e.code in ['0151', '0152']: # Servicio no disponible raise self.retry(exc=e) else: # Error de validación, no reintentar log_error(factura_id, e)

Beneficio:

Si SUNAT cae a las 10:00 AM y vuelve a las 10:30 AM, Celery reintenta automáticamente todas las facturas pendientes. Sin intervención manual.

3. Tareas Programadas (Cron Jobs)

# Cierre automático de cajas a medianoche @celery_app.task def cerrar_cajas_automaticamente(): cajas_abiertas = db.query(Caja).filter( Caja.estado == "abierta" ).all() for caja in cajas_abiertas: cerrar_caja(caja.id) enviar_reporte(caja.supervisor_id) # Programar para las 00:00 todos los días celery_app.conf.beat_schedule = { 'cerrar-cajas-diario': { 'task': 'cerrar_cajas_automaticamente', 'schedule': crontab(hour=0, minute=0), }, 'verificar-facturas-pendientes': { 'task': 'verificar_estados_sunat', 'schedule': crontab(minute='*/5'), # Cada 5 min } }

4. Reportes Pesados

# Generar reporte anual de ventas (tarda 30-60 segundos) @celery_app.task def generar_reporte_anual(user_id, year): data = consulta_pesada_bd(year) # 15 segundos graficos = generar_graficos(data) # 10 segundos pdf = crear_pdf(data, graficos) # 10 segundos url = subir_a_storage(pdf) # 5 segundos notify_user(user_id, f"Reporte {year} listo: {url}")

Sin Celery: Usuario hace clic en "Generar reporte" → espera 60 segundos mirando spinner → se frustra y cierra la pestaña.

Con Celery: Clic → respuesta inmediata "Generando..." → sigue trabajando → recibe notificación cuando está listo.

¿Cuándo NO necesitas Celery?

Solo si NO tienes:

  • ❌ Facturación electrónica (SUNAT/OSE)
  • ❌ Reportes que tardan más de 5 segundos
  • ❌ Tareas programadas (cron jobs)
  • ❌ Necesidad de reintentos automáticos
  • ❌ Envío de emails masivos

Pero para facturación electrónica, Celery es crítico porque:

  • SUNAT puede tardar 1-5 segundos en responder
  • SUNAT puede estar caído (necesitas reintentos)
  • No puedes bloquear la cajera esperando respuesta

✅ Conclusión Celery

Para facturación electrónica y sistema POS, Celery es altamente recomendado.

Costo adicional: $0 | Beneficio: 27x más rápido + reintentos automáticos

💡 Recomendación Final

Implementar Redis y Celery desde el inicio.

Razones:

  1. Sesión única es un requerimiento obligatorio (necesita Redis)
  2. Facturación electrónica sin bloqueo es crítico para POS (necesita Celery)
  3. Costo adicional: $0-5/mes (mínimo vs beneficio)
  4. Migrar después es mucho más trabajo
  5. Experiencia del usuario es profesional vs amateur

Si no los usas: Tu sistema funcionará, pero será lento, propenso a errores, y frustrante para los usuarios. No podrás garantizar sesión única ni manejar caídas de SUNAT.
